# GB28181 Protocol Support Design

**Date**: 2026-04-04
**Status**: Draft
**Scope**: Add GB28181 video surveillance protocol with publish, play, cluster-push, cluster-pull

## Overview

GB28181 (GB/T 28181) is a Chinese national standard for video surveillance networking. It uses SIP signaling for device management and session control, with RTP carrying MPEG-PS (Program Stream) payloads for media transport.

This design adds GB28181 as a new protocol in liveforge, with two new modules (SIP + GB28181) and a new PS muxer/demuxer library.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| SIP role | Server only (upper platform) | Covers 90% of use cases. Cascading can be added later. |
| Media transport | UDP + TCP | UDP is default; TCP for firewall/cross-network scenarios. |
| Feature scope | Full: live, playback, PTZ, catalog, alarm | Complete GB28181 compliance. |
| SIP library | `emiago/sipgo` | Cleanest API, leanest deps, active maintenance, used by SRS project. |
| PS parser | Built-in `pkg/muxer/ps/` | Consistent with existing muxer pattern. PS is well-documented (ISO 13818-1). |
| Device store | In-memory + optional file dump | Consistent with StreamHub pattern. Devices re-register on restart. |
| Architecture | Split module (SIP + GB28181) | SIP layer is reusable; GB28181 owns business logic. |

## Architecture

```
pkg/sip/              pkg/muxer/ps/
  (SIP helpers)         (PS demux/mux)

module/sip/           module/gb28181/
  (SIP transport)       (GB28181 business logic)
       ^                      |
       |  SIPService iface    |
       +----------------------+

module/cluster/
  transport_gb.go     (GB28181 cluster relay)
```

### Separation of Concerns

- **`pkg/sip/`** -- Thin convenience types around sipgo. Not reimplementing SIP.
- **`module/sip/`** -- Owns sipgo UA, UDP/TCP listener, transaction management. Exposes `SIPService` interface for other modules. Knows nothing about GB28181.
- **`module/gb28181/`** -- All GB28181 business logic: device registry, media sessions, catalog/PTZ/alarm. Consumes `SIPService`. Emits core events (`EventPublish`, `EventSubscribe`, etc.).
- **`pkg/muxer/ps/`** -- Pure library. PS demux/mux. No module dependencies.
- **`module/cluster/transport_gb.go`** -- `RelayTransport` implementation for GB28181 PS relay.

### Module Interaction

```
module/sip (owns sipgo)
    |
    +-- OnRegister(handler)  --> module/gb28181 handles device registration
    +-- OnInvite(handler)    --> module/gb28181 handles incoming INVITE
    +-- OnMessage(handler)   --> module/gb28181 handles catalog/PTZ responses
    +-- OnBye(handler)       --> module/gb28181 handles session teardown
    +-- SendRequest(req)     <-- module/gb28181 sends INVITE/MESSAGE/SUBSCRIBE
```

Inter-module communication uses **constructor injection** — no service locator. The SIP module exposes its `SIPService` via a method, and `main.go` passes it to the GB28181 module constructor:

```go
// cmd/liveforge/main.go
if cfg.SIP.Enabled {
    sipMod = sipmod.NewModule()
    s.RegisterModule(sipMod)
}

if cfg.GB28181.Enabled {
    s.RegisterModule(gb28181mod.NewModule(sipMod.Service()))
}
```

This provides compile-time type safety and makes the dependency explicit.

**Lifecycle timing:** `sipMod.Service()` is called in `main.go` **before** `s.Init()`. This works because `NewModule()` pre-allocates the `SIPService` struct (a concrete type, not nil). The struct holds method handler registrations (slices of callbacks) and a sipgo UA pointer that starts as nil. When `Init()` runs later, it populates the sipgo UA and starts listeners in place. The GB28181 module's `Init()` then calls `OnRegister(handler)` etc. to register its handlers on the already-allocated service struct. By the time any SIP message arrives, both modules are fully initialized.

```go
// module/sip/module.go
type Module struct {
    service *Service  // pre-allocated in NewModule()
}

func NewModule() *Module {
    return &Module{
        service: newService(),  // struct allocated, UA field is nil
    }
}

func (m *Module) Service() SIPService { return m.service }

func (m *Module) Init(s *core.Server) error {
    // Now create sipgo UA and populate m.service internal fields
    m.service.init(s.Config().SIP)  // starts listeners, sets UA
    return nil
}
```

### Module Registration Order

Modules must be registered in this order in `cmd/liveforge/main.go`:

1. **auth** — hooks must be in place before protocol modules
2. **rtmp, rtsp, srt, webrtc, httpstream** — protocol modules
3. **sip** — SIP transport, must be before gb28181
4. **gb28181** — depends on SIP service, must be before cluster
5. **notify** — WebSocket handler must be registered before API
6. **cluster** — signaling handlers must be registered before API; GBTransport registered here
7. **api** — snapshots all HTTP handlers
8. **record, metrics** — independent

## SIP Module (`module/sip/`)

### SIPService Interface

```go
type SIPService interface {
    OnRegister(handler RegisterHandler)
    OnInvite(handler InviteHandler)
    OnBye(handler ByeHandler)
    OnMessage(handler MessageHandler)
    OnSubscribe(handler SubscribeHandler)
    OnNotify(handler NotifyHandler)

    SendRequest(ctx context.Context, req *sip.Request) (*sip.Response, error)
    SendInvite(ctx context.Context, req *sip.Request) (*InviteTransaction, error)

    LocalAddr() string
    ServerID() string
    Domain() string
}
```

### Module Details

- `Name()` returns `"sip"`
- `Hooks()` returns empty `[]HookRegistration` — SIP is pure transport, no lifecycle hooks
- `Service()` returns the `SIPService` implementation for injection into dependent modules
- `Init()` creates sipgo UA, starts UDP+TCP listeners, sets up method dispatch
- `Close()` gracefully stops listeners and drains in-flight transactions

### File Structure

```
module/sip/
  module.go        -- core.Module: Init starts sipgo, Close shuts down; Service() accessor
  server.go        -- SIP listener setup (UDP+TCP), sipgo Server/Client init
  service.go       -- SIPService interface definition
  dispatch.go      -- Routes incoming SIP methods to registered handlers
  auth.go          -- Digest authentication helper
  transaction.go   -- InviteTransaction wrapper for INVITE/ACK/BYE lifecycle
```

### Device Registry

```go
type Device struct {
    DeviceID      string
    RemoteAddr    string
    Transport     string
    RegisteredAt  time.Time
    LastKeepalive time.Time
    Status        DeviceStatus
    Channels      map[string]*Channel
}

type Channel struct {
    ChannelID    string
    Name         string
    Manufacturer string

    PTZType      int
    Latitude     float64
    Longitude    float64

```

Offline detection: background goroutine checks `LastKeepalive` against `keepalive.timeout` (default 3x interval). Marks device offline and closes active sessions.

File dump: periodic + on shutdown to `dump_file`. Restore on startup for faster cold-start.

### SIP Handler Flows

**REGISTER:**
1. Auth challenge if enabled (401 -> re-REGISTER with credentials)
2. Parse Contact, Expires
3. If Expires=0 -> unregister, else register/refresh
4. Auto-query catalog after first registration

**Incoming INVITE (device push / publish):**
1. Parse SDP offer -> extract RTP port, codec info
2. Allocate local RTP port
3. Create Stream via StreamHub.GetOrCreate()
4. Create GB28181Publisher, set on Stream
5. Reply 200 OK with SDP answer (our RTP listen port)
6. Start RTP receive goroutine

**BYE:**
1. Find session by Call-ID/dialog
2. Close media session (stop RTP, remove publisher)
3. Reply 200 OK

**MESSAGE:**
1. Parse XML body: Keepalive / Catalog / Alarm / DeviceInfo
2. Dispatch to appropriate handler
3. Reply 200 OK

### Media Session State Machine

```
              INVITE received/sent
    IDLE --------------------------> INVITING
                                        |
                                  200 OK + ACK
                                        |
                                        v
                                    STREAMING
                                        |
                                   BYE / timeout
                                        |
                                        v
                                     CLOSED
```

```go
type MediaSession struct {
    ID          string          // SIP Call-ID
    DeviceID    string
    ChannelID   string
    StreamKey   string
    Direction   Direction       // Inbound / Outbound
    LocalPort   int
    RemoteAddr  *net.UDPAddr
    Transport   string
    State       SessionState
    Publisher   *GB28181Publisher
    Stream      *core.Stream
    SSRC        uint32        // GB28181 SSRC (consistent with pion/rtp and existing codebase)
}
```

### Stream Key Mapping

```
Live:     gb28181/{channelID}
Playback: gb28181/{channelID}/playback/{startUnix}
```

`startUnix` is the playback start time as Unix timestamp (seconds). Example: `gb28181/34020000001320000001/playback/1743760800`.

Allows playback via all existing protocols:
- `rtmp://host/gb28181/34020000001320000001`
- `http://host/gb28181/34020000001320000001.m3u8`
- `WHEP POST /whep/gb28181/34020000001320000001`

### Play (Platform Invites Camera)

```go
func (m *Module) InvitePlay(channelID string) (*MediaSession, error)
```

1. Find device/channel in registry
2. Allocate local RTP port
3. Generate SSRC (GB28181 rule: domain prefix + sequence)
4. Build SDP offer (recvonly, our listen port, SSRC)
5. Send INVITE via SIPService.SendInvite()
6. Wait for 200 OK, parse SDP answer
7. Send ACK
8. Create Stream, GB28181Publisher, start RTP receive
9. Return session

### Playback

Same as play but with time range in SDP:
- `t=<start_ntp> <end_ntp>`
- `u=<channelID>:3` (playback type)
- Supports playback control via INFO method with MANSRTSP body (PLAY, PAUSE, Scale)

### Catalog Query

Auto-triggered after device registration. Sends MESSAGE with XML body, parses multi-part catalog response, populates channels in DeviceRegistry.

### PTZ Control

Sends MESSAGE with XML body containing 8-byte hex-encoded PTZ command per GB28181 Annex A. Supports directional control (up/down/left/right), zoom, and combined movements.

### Alarm Subscription

SUBSCRIBE to device alarm events, process NOTIFY messages, emit `EventGB28181Alarm` via EventBus for webhook/websocket forwarding.

## PS Muxer/Demuxer (`pkg/muxer/ps/`)

### Format

```
RTP Packet
  -> PS Pack
    -> Pack Header    (00 00 01 BA) -- SCR, mux rate
    -> System Header  (00 00 01 BB) -- optional, on keyframes
    -> PES Packet(s)  (00 00 01 E0) -- video: H.264/H.265
                      (00 00 01 C0) -- audio: G.711/AAC
```

### Demuxer

```go
type Demuxer struct { ... }
func NewDemuxer() *Demuxer
func (d *Demuxer) Feed(data []byte) ([]*avframe.AVFrame, error)
```

Handles: fragmented PES across RTP packets, codec auto-detection, SPS/PPS extraction for SequenceHeader, PTS/DTS conversion to milliseconds.

### Muxer

```go
type Muxer struct { ... }
func NewMuxer() *Muxer
func (m *Muxer) Pack(frame *avframe.AVFrame) ([]byte, error)
```

Generates complete PS packs. System header included on keyframes only.

## GB28181 Publisher & RTP Receiver

### Data Flow

```
Camera --RTP/UDP--> RTPReceiver.Run()
                        |
                   reorderBuffer (UDP only)
                        |
                   publisher.FeedRTP(pkt)
                        |
                   ps.Demuxer.Feed(payload)
                        |
                   []*avframe.AVFrame
                        |
                   stream.WriteFrame(frame)
                        |
              +---------+----------+
              v         v          v
           RTMP sub  HLS muxer  WebRTC sub
```

### RTP Reorder Buffer (UDP)

```go
type reorderBuffer struct {
    buf      map[uint16]*rtp.Packet
    nextSeq  uint16
    maxDelay uint16  // default 50
}
```

Flushes consecutive packets from nextSeq. Skips gaps exceeding maxDelay.

### TCP Mode

RFC 4571: 2-byte length prefix + RTP packet. No reorder needed (TCP is ordered).

### RTP Marker Bit

`M=1` indicates end of PS pack / access unit. Demuxer accumulates payload until marker, then processes complete PS pack.

## Cluster Transport (`module/cluster/transport_gb.go`)

### GBTransport

```go
type GBTransport struct { ... }
func (t *GBTransport) Scheme() string { return "gb28181" }
func (t *GBTransport) Push(ctx, targetURL, stream) error
func (t *GBTransport) Pull(ctx, sourceURL, stream) error
```

Uses SDP-over-HTTP signaling (same pattern as existing RTPTransport) with PS encapsulation.

**Push:** Stream -> RingBuffer -> AVFrame -> PS Mux -> RTP -> UDP to remote
**Pull:** Remote -> RTP/UDP -> PS Demux -> AVFrame -> stream.WriteFrame()

### Signaling Endpoints

```
POST /api/relay/gb/push?stream={key}
POST /api/relay/gb/pull?stream={key}
```

### Usage

```yaml
cluster:
  forward:
    enabled: true
    targets:
      - "gb28181://remote-node:8090/live/cam01"
  origin:
    enabled: true
    servers:
      - "gb28181://origin-node:8090"
```

**Note:** GB28181 cluster relay requires the cluster module to be enabled (`forward.enabled` or `origin.enabled`). The `GBTransport` is registered inside the cluster module's `Init()`. If cluster is disabled, `gb28181://` URLs in forward/origin targets will not resolve. This is consistent with how all other cluster transports (RTMP, SRT, RTSP, RTP) work.

### Port Allocation

The codebase has two independent port allocators with different APIs:
- `module/cluster/transport_rtp.go` `portAllocator`: single-port allocation, string range `"20000-20100"`
- `module/rtsp/transport.go` `PortManager`: port-pair allocation (even RTP + odd RTCP), `[]int` range

GB28181 needs **port-pair allocation** (standard RTP: even port for RTP, odd for RTCP). Extract both into a unified shared package:

```
pkg/portalloc/
  portalloc.go      -- PortAllocator with AllocatePair()/Allocate()/Free()
  portalloc_test.go
```

```go
// pkg/portalloc/portalloc.go

type PortAllocator struct { ... }

func New(minPort, maxPort int) *PortAllocator

// Allocate returns a single available port.
func (pa *PortAllocator) Allocate() (int, error)

// AllocatePair returns an even/odd port pair for RTP+RTCP.
func (pa *PortAllocator) AllocatePair() (rtpPort, rtcpPort int, err error)

// Free returns port(s) to the pool.
func (pa *PortAllocator) Free(ports ...int)
```

Consumers to update:
- `module/rtsp/transport.go` PortManager → use `pkg/portalloc`
- `module/cluster/transport_rtp.go` portAllocator → use `pkg/portalloc`
- `module/gb28181/` → use `pkg/portalloc` with `AllocatePair()`
- `module/cluster/transport_gb.go` → use `pkg/portalloc`

All port range configs use `[]int` format `[min, max]` for consistency with existing `RTSPConfig.RTPPortRange`. The `ClusterGBConfig.PortRange` is `[]int`, not string.

## Error Handling

### SIP Auth Failures

When digest auth is enabled:
1. First REGISTER without credentials -> 401 Unauthorized with WWW-Authenticate (nonce, realm)
2. Re-REGISTER with correct credentials -> 200 OK, proceed to registration
3. Re-REGISTER with wrong credentials -> 403 Forbidden, log warning, drop
4. Replay attack (stale nonce) -> 401 with `stale=true`, allow client to retry with fresh nonce

### Device Offline Transition

When `LastKeepalive` exceeds `keepalive.timeout`:
1. Mark device `Status = Offline` in registry
2. For each active MediaSession bound to this device:
   a. Send BYE via SIPService (best-effort, don't block on response)
   b. Call `stream.RemovePublisher()` to stop frame distribution
   c. Stream's `no_publisher_timeout` handles eventual cleanup via existing StreamHub logic
3. Emit `EventPublishStop` for each affected stream (uses existing event, not a GB28181-specific event)
4. Log at WARN level with device ID and channel count

### Graceful Shutdown

When `module/gb28181` `Close()` is called:
1. Send BYE to all active media sessions (best-effort, 2s timeout)
2. Close all RTP receivers (stops UDP/TCP read loops)
3. Dump device registry to file (if `dump_file` configured)
4. Deregister SIP method handlers

## Event Types

GB28181-specific events use the existing `EventContext.Extra` map rather than adding protocol-specific event types to core. This keeps core protocol-agnostic:

```go
// Device online: emit EventPublish with Extra["gb28181_device_id"]
// Device offline: emit EventPublishStop with Extra["gb28181_device_id"]
// Alarm: emit via notify module webhook with Extra["gb28181_alarm_method", "gb28181_alarm_type"]
```

For alarm notifications specifically, the GB28181 module calls the notify module's webhook/websocket directly via the EventBus using a generic event approach, avoiding new event type constants in core.
